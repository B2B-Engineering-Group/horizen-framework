## demo_mono

Шаблон монорепозитория (back + front) в одном корне.

### Договоренности и конвенции

- **Контроллеры**: актуальный способ описания контроллеров — через `controllers.js`. Описание “через папки” — легаси-подход.

- **API пути**: предпочитаем простые и плоские пути без лишнего дробления на подпути через `/` (используем редко, только для особых случаев). Примеры:
  - `/api/getSomething`
  - `/api/updateSomething`
  - `/api/createSomething`
  - `/api/getSomethingList`

- **Mongo `_id`**: для сущностей, где нужен новый `_id`, предпочитаем `uuidv4`. Классические Mongo `ObjectId` стараемся никогда не использовать.

- **Frontend компоненты**: новые компоненты пишем цельными файлами в стиле `components/SomeComponent.jsx`. Стараемся не дробить на множество мелких файлов и держать компонент атомарным на уровне архитектурной сущности.

- **Тонкие контроллеры / атомарные сервисы**: контроллеры держим максимально тонкими — они только вызывают функции сервисов. Сервисы, наоборот, стараемся держать большими и атомарными (как архитектурные сущности). Провайдеры в отдельные папки выделяем только при реальной необходимости — даже если сущность “просится” в отдельный провайдер, часто оставляем её в сервисе разбивая там (обеспечивая удобную передачу контекста для AI).

- **Процессы и демоны**: backend `server` процесс классически используем для лёгких API операций, он потенциально реплицируем (в зависимости от проекта). Тяжёлые операции и демоны выносим в отдельные hidden-процессы. Сами демоны и их цепочки описываются в сервисе, но вызов происходит из соответствующего процесса. Цепочки демонов часто строятся на базе Mongo-оркестрации: документ — это “таск”, а отдельные поля в нём — флаги для тех или иных демонов.

- **`module.schema.json`**: генерируется автоматически после запуска backend (после `return` в `horizen.init`).

### Запуск локально

- **Back**: `horizen-run server`

- **Front**: `horizen-run dev`
